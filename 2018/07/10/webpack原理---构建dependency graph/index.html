webpack原理—构建Dependency Graph

对于webpack如何分析依赖，构建依赖树，我之前一直单纯地以为，读取入口文件，然后正则查找require、import等来查找依赖，但是细想的话，这种方式肯定是不合理的，因为如果注释中import或者require的话，或者变量命名中带有这些的话，也会被匹配进去，肯定是不符合要求的。

那么，webpack究竟如何识别依赖，并构建Dependency Graph的呢？

下面分三个步骤来研究。

1、webpack打包出来的文件到底长什么样

之前一直用webpack打包，但是很少关注webpack打包出来的文件长什么样。下面通过一个简单的例子来看一下：

下面是打包出来的文件：

总体来看，就是一个大的IIFE函数。

这个函数体，就是定义了一个__webpack_require__函数，然后定义了几个这个函数的‘静态方法’，最后就是return出调用这个函数的结果。

就是这么一个简单的函数，就是函数名特别奇葩…

OK，来看一下这个函数，

  • 首先，判断缓存中是否存在，moduleId就是文件的路径。如果有的话，直接返回。

  • 创建一个新的module对象，并且存在缓存中。

  • 执行参数对象对应id的方法，并把module, module.exports, __webpack_require__作为参数传入，然后this指向module.exports。

Nodejs模块系统的实现原理也是上面这样。

接下来看看传进去的参数（暂起名叫args），它就是一个对象，这个对象有两个方法，'./a.js'和'./index.js'。

我现在还不明白为什么，这个两个函数的定义，要用()包裹起来…

./a.js ：

./index.js：

先看/indexjs的eval：

  • __webpack_require__.r(__webpack_exports__);如果支持Symbol，就给__webpack_exports__对象添加两个属性Symbol.toStringTag、 __esModule ，其值分别为'Module'和true。
  • var _a__WEBPACK_IMPORTED_MODULE_0__ = __webpack_require__("./a.js")递归调用__webpack__require__，所以执行到’./a.js’方法下的eval函数中去了。
    • __webpack_require__.r(__webpack_exports__) 跟上面一样
    • __webpack_require__.d(__webpack_exports,"default",fucntion(){return a})设置__webpack_exports__的default属性的getter为function(){return a}
    • __webpack_require__.d(__webpack_exports,"f",fucntion(){return f})设置__webpack_exports__的f属性的getter为function(){return f}
    • function a () {console.log('a')}
    • function f () {console.log('f')}
  • Object(_a__WEBPACK_IMPORTED_MODULE_0__["default"])() 执行a()

所以，参数对象args里扁平的存储了我们所有的依赖模块，并且模块内容用eval形式包裹起来。而每个模块的__webpack__require__()就表明了自己所依赖的模块。所以我们的js开始执行的时候，就会首先从入口开始，根据每个模块自己的依赖关系，深度优先遍历，把我们程序中可能用到的所有模块缓存到installedModules里。id值就是文件的路径，并且对于同一个模块，只缓存一次。

有一点需要注意，我们index.js中，并没有使用到函数f，但是我们在打包的时候，依然把函数f打进来了。当然这里的原因是我使用的是webpack的development模式，而不是production模式。在production模式下，函数f是不会被打进来的。这就是webpack的tree-shaking。但是tree-shaking也不能完完全全地使我们的代码纯粹。https://juejin.im/post/5a4dc842518825698e7279a9这篇文章对tree-shaking有很好的解释。

2、分析每个模块自己依赖的基础–AST

正如开头所说，通过正则方式匹配’import’或者’require’肯定是不合理的，所以我们采用先把代码转成AST，然后再遍历分析。

AST，抽象语法树，是源码抽象结构的树状表现形式。是我们对源码进行一些操作的基础，babel、webpack在静态解析我们代码的时候，用的就是AST，通过http://astexplorer.net/#/gist/5579c7f7b371a6f7781f974c9aa1bb6a/238c6f5a96217c0e3b403e33e9de2483da50b0fe可以方便地查看我们代码的AST。

通过AST分析依赖并构建依赖树

如上面AST的图，我们可以通过判断type是否是’ImportDeclaration’的方式来找到当前文件所有的import形式的依赖（a、b、c），然后再挨个读取依赖文件（a->b->c），分别构建当前文件AST，再获得当前文件的所有依赖…，这样就构成了Dependency Graph。这种遍历方式我理解的是广度优先遍历的方式（网上好多人说是深度优先遍历，我用的是4.15.1）。
